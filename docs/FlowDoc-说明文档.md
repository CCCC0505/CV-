# FlowDoc 当前实际与 PRD 差距说明文档

## 1. 文档目的

这份文档不是功能宣传稿，而是用于说明 **当前 Demo 实际落地情况、与 PRD 的差距、已做的技术取舍、以及后续还需要补多少工作量** 的对齐文档。

建议把这份文档理解为 4 个用途：

1. 对内统一口径，避免把“可演示能力”说成“完整产品能力”。
2. 对外答辩时，明确哪些能力已经可跑通，哪些还属于 Demo 级实现。
3. 对照 PRD 判断目前完成度，而不是只看页面是否“看起来有”。
4. 为后续 GitHub 提交、录屏展示、答辩问答提供依据。

---

## 2. 这份说明文档应该包含什么

如果这份文档要真正有用，至少应该覆盖下面几类信息：

1. 产品目标与 PRD 主线到底是什么。
2. 当前代码已经实现了哪些页面、哪些接口、哪些 AI 能力。
3. 哪些能力是“真实调用千问”的，哪些能力带有 fallback 或规则兜底。
4. 哪些功能只是 Demo 级可演示，还没有达到完整产品化。
5. 哪些 PRD 点已经完成，哪些只完成了第一版，哪些还没有完成。
6. 当前最影响评审感知的问题是什么。
7. 如果继续做，还需要多少时间，优先级应该怎么排。

当前旧版说明文档的问题就在于：

1. 有完成度结论，但不够像“差距说明”。
2. 有功能罗列，但没有把“已落地 / 部分落地 / 未落地”分层写清楚。
3. 没有把 AI 真实接入与本地兜底逻辑说明白。
4. 没有把“为什么现在能演示，但还不算完全做完”讲透。

## 3. 项目目标与 PRD 主线

根据 [Demo-PRD.md](D:/CVTE极客Π/Demo-PRD.md)，FlowDoc 的核心不是“做一个 Markdown 编辑器”，而是证明：

1. 文档可以从静态记录工具升级成 AI 原生的业务决策空间。
2. 同一份原文可以被不同岗位以不同视角审阅，但原文本身不被改写。
3. 文档除了正文，还能承载任务、摘要、代码、数据、版本决策、知识关联和 AI 对话。
4. 文档不是一次性产物，而是会随着修改、审阅和数据变化持续演化。

因此，PRD 的主线不是“编辑器功能越多越好”，而是这条闭环是否成立：

`配置工作流 -> 导入文档 -> AI 分析 -> 进入编辑 -> 角色审阅 -> 版本确认 -> 决策溯源 -> 无界能力扩展`

从这个角度看，当前项目已经形成主线，但距离 PRD 理想态还有明显差距，主要差在“深度、稳定性、协作真实性、UI 完成度”四个方向。

## 4. 当前实际落地概览

### 4.1 页面结构

当前已实现 3 个主页面：

1. `/` 首页
2. `/setup` 配置与分析页
3. `/editor` 编辑与审阅页

当前页面拆分符合前期方案，也基本符合 PRD 中“先配置、再分析、后进入编辑”的流程要求。

### 4.2 后端接口

当前已经落地的主要接口如下：

1. `GET /api/health`
2. `GET /api/presets`
3. `GET /api/presets/{preset_id}`
4. `POST /api/analyze`
5. `POST /api/review-enrich`
6. `POST /api/chat`
7. `POST /api/code-lab`
8. `POST /api/viz-assist`
9. `POST /api/doc-relations`
10. `POST /api/fold-plan`
11. `POST /api/version-summary`

这说明当前项目已经不是一个纯前端壳子，而是已经形成了 **多能力拆分的后端服务结构**。

### 4.3 AI 服务拆分

当前 AI 能力不是全塞在一个文件里，而是按功能拆分成多个服务文件：

1. `qwen.py`：工作流分析
2. `qwen_review.py`：角色审阅增强
3. `qwen_chat.py`：侧边栏对话
4. `qwen_code.py`：代码实验辅助
5. `qwen_viz.py`：数据可视化辅助
6. `qwen_relation.py`：文档关联推荐
7. `qwen_fold.py`：角色视角折叠规划
8. `qwen_version.py`：版本摘要与自写结论

这一点是当前版本非常明确的技术优点，因为它避免了“所有 AI 逻辑绑死在一次大请求里”的问题，也符合你后期反复强调的架构方向。

## 5. 当前实际完成情况

下面这部分按“已完成 / 部分完成 / 未完成”来讲，而不是只写“有这个功能”。

### 5.1 已完成的主链路能力

#### A. 三页式主流程

已完成：

1. 首页作为入口页。
2. 配置页完成工作流选择、文档上传或预置文档选择。
3. 点击开始分析后，成功才出现进入编辑区按钮。
4. 编辑页与配置页分离，不再混放上传和工作流组件。

结论：

这条主路径已经具备 Demo 级完整可演示性。

#### B. 原文保留机制

已完成：

1. 编辑区始终保留完整原文。
2. 角色切换不会改写正文。
3. 折叠只作用于视图层，而不是改正文内容。
4. 任务安排、角色摘要、关联文档等都在正文之外展示。

结论：

这是当前项目最稳定、最符合 PRD 精神的一条主线，也是答辩时最值得强调的点之一。

#### C. 配置页分析闭环

已完成：

1. 支持默认工作流和自定义工作流。
2. 支持上传 `.md/.txt` 和选择预置文档。
3. 点击开始分析后，请求后端分析接口。
4. 分析成功后展示岗位流转图。
5. 分析结果通过 `sessionStorage` 传入编辑页。

结论：

功能闭环已成立，但“分析质量”和“图的精致程度”仍有提升空间。

#### D. 编辑页角色审阅

已完成：

1. 顶部角色切换。
2. 编辑视图与角色审阅视图切换。
3. 角色审阅时可按角色进行段落折叠。
4. 不相关段落用“点击展开 X 个折叠段落”形式提示。
5. 相关段落带有高亮审阅样式。
6. 审阅确认后支持切换到工作流中的下一个角色。

结论：

这部分已经具备 PRD 中“角色视角不改原文”的核心特征，但审阅体验仍偏 Demo 化。

#### E. 版本时间轴与决策溯源

已完成：

1. 编辑后需点击“确认更改”才记版本。
2. 确认更改后才触发版本摘要和自写结论请求。
3. 左侧抽屉支持摘要与时间轴切换。
4. 时间轴会记录版本摘要、自写结论、受影响角色和关键变化。
5. 支持恢复到某一版本并再次确认形成新版本。

结论：

这条能力已经具备“决策溯源”的基础形态，是当前 Demo 的第二个强亮点。

#### F. 侧边栏 AI 对话

已完成：

1. 左侧预留为 AI 对话区域，而不是移到页面顶部。
2. 用户左键选中文本后可加入对话上下文。
3. 支持角色预设，如专业产品经理、专业投资人、专业工程师、专业数据分析师。
4. 对话内容在左侧面板持续展示。

结论：

这部分已经符合“边编辑边问”的交互方向，但仍属于轻量问答，不是复杂 Agent 工作流。

#### G. 无界能力扩展

已完成第一版：

1. 代码实验区
2. 数据可视化区
3. 关联文档区

结论：

这三块已经放进最终编辑页顶部，并随角色切换联动展示，但深度仍明显低于 PRD 想象态。

## 6. 当前哪些能力是真 AI，哪些是兜底演示态

这是这份文档里必须讲清楚的一部分。

### 6.1 真实依赖千问 API 的能力

在 `DASHSCOPE_API_KEY` 配置正常时，以下能力会真实调用千问：

1. 工作流分析 `qwen.py`
2. 角色审阅增强 `qwen_review.py`
3. 左侧 AI 对话 `qwen_chat.py`
4. 代码实验解释 `qwen_code.py`
5. 数据可视化辅助 `qwen_viz.py`
6. 文档关联推荐 `qwen_relation.py`
7. 角色折叠规划 `qwen_fold.py`
8. 版本自写结论 `qwen_version.py`

### 6.2 带兜底能力的模块

以下模块在 AI 调用失败、JSON 不合法、网络超时、返回结构异常时，会进入 fallback 或修复路径：

1. `qwen_code.py`
2. `qwen_viz.py`
3. `qwen_relation.py`
4. `qwen_fold.py`
5. `qwen.py`
6. `qwen_review.py`
7. `qwen_version.py`

说明：

1. `analyze/review/version/chat` 这类关键模块会尽量先做 JSON 修复，再返回。
2. `code/viz/relation/fold` 除了 JSON 修复，还准备了更明显的本地兜底结果，保证页面至少可演示。

### 6.3 哪些能力本质上还是 Demo 级

1. Python/C/Java 代码运行不是真执行，而是 AI 解释 + 伪运行结果。
2. 数据可视化更多是“表格 + 预置图形态”，不是真正完整 BI 分析。
3. 文档关联当前更偏“AI 推荐 + 只读预览 + 关系图展示”，不是深层知识网络。
4. 角色折叠当前还是基于 AI 提示和关键词映射的轻量策略，不是稳定的语义级理解引擎。

结论：

当前版本是“AI 驱动的可演示原型”，不是“生产级协作平台”。

## 7. 与 PRD 的主要对照

下面按 PRD 核心模块来做差距判断。

### 7.1 工作流分析

PRD 目标：

1. 基于文档和工作流生成岗位流转图。
2. 给出具体的岗位任务、重点关注项、工作摘要、任务安排。

当前情况：

1. 已接入千问分析。
2. 已有结构化 JSON 校验与修复。
3. 已能渲染岗位流转图和任务安排。
4. 但分析结果的“细致程度”和“业务具体性”不稳定，仍容易出现模板感。

差距判断：

1. 功能已落地。
2. 质量未完全达到 PRD 预期。
3. 更像“能跑通”，还不是“能稳定给出很强结果”。

### 7.2 角色视图与折叠

PRD 目标：

1. 保留原文。
2. 不同岗位只看与自己相关的部分。
3. 与当前角色无关内容折叠。
4. 提供角色摘要和审阅支持。

当前情况：

1. 原文保留已达成。
2. 审阅视图、折叠提示、高亮、角色摘要已达成。
3. 折叠规划已拆独立服务。
4. 但折叠准确率和交互精度还有明显提升空间。

差距判断：

1. 核心理念已完成。
2. 体验质量属于第一版。

### 7.3 决策溯源与版本时间轴

PRD 目标：

1. 修改后形成版本记录。
2. 版本确认后再调用 AI 做变更总结与自写结论。
3. 用于决策回顾与认知演化展示。

当前情况：

1. 已支持确认更改后生成新版本。
2. 已支持 AI 生成 change summary 与 self conclusion。
3. 已支持时间轴展示和版本恢复。

差距判断：

1. 主功能已完成。
2. 视觉表达、历史轨迹感、版本差异高亮还未达到 PRD 理想态。

### 7.4 AI 对话

PRD 目标：

1. 选中文本后加入对话。
2. 在侧栏实时提问。
3. 有角色预设。

当前情况：

1. 已完成。
2. 交互细节已经比早期版本好很多。
3. 但仍是面向单轮或轻多轮问答，不是复杂工作流型智能助手。

差距判断：

1. 已满足演示需求。
2. 未达到深度 Copilot 或决策助理级别。

### 7.5 代码实验

PRD 目标：

1. 产品经理、工程师页顶部有代码实验区。
2. HTML/JS 可运行。
3. Python/C/Java 可以做解释和补全。

当前情况：

1. 已完成。
2. HTML/JS 可浏览器预览。
3. 其余语言走 AI 解释 + 伪运行。

差距判断：

1. 与计划一致。
2. 这是“受控降级实现”，不算偏差，但要在答辩里讲清楚。

### 7.6 数据可视化

PRD 目标：

1. 数据分析员 / CEO / 运营可以看到数据可视化区域。
2. 默认有预置图表。
3. 支持链接数据源。

当前情况：

1. 已完成预置表格与预置图。
2. 已有“当前未链接数据源，采取预置数据显示”的说明。
3. 上传数据源已接入，但目前 CSV 支持非常轻量，只适合一行或一列这种低复杂度输入。

差距判断：

1. 展示能力已落地。
2. 真正的数据接入能力还很浅。

### 7.7 关联文档

PRD 目标：

1. 推荐关联文档。
2. 用户确认关联。
3. 展示关系类型与关系图。
4. 能点开查看关联文档。

当前情况：

1. 已有后台预置 md 文档。
2. 已有候选文档选择、确认关联、关系图、只读预览浮窗。
3. 关联后能在编辑区上方显示关联文档入口。

差距判断：

1. 该能力已经可演示。
2. 但当前创新深度偏弱，更像“只读预览 + 关系展示”，还不是强知识网络。

## 8. 当前版本最主要的差距

如果只讲最关键的问题，当前与 PRD 的差距主要有 5 个：

### 8.1 AI 结果质量不够稳定

虽然分析、审阅、折叠、版本总结都已经接入千问，但结果质量仍受以下因素影响：

1. 文档长度限制
2. prompt 压缩程度
3. 模型返回结构不稳定
4. timeout 风险

这意味着“能返回结果”不等于“总能返回足够具体、足够惊艳的结果”。

### 8.2 无界能力更多是扩展展示，不是深度生产能力

代码实验、可视化、关联文档都已经有，但目前仍偏向：

1. 演示导向
2. 轻量交互
3. 有视觉、有闭环，但深度有限

这对 Demo 是够用的，但如果严格按 PRD 的“重新定义文档”口径来看，还差一层更强的业务穿透力。

### 8.3 协作真实性不够强

当前版本本质上还是：

1. 单浏览器会话
2. 单用户操作
3. AI 模拟多角色协同

它不是实时多人同步，也没有真实权限、锁定、审批流。

### 8.4 UI 完成度还没有完全收口

虽然现在比早期版本已经好很多，但从比赛成品角度看，仍有以下问题：

1. 个别卡片和浮层仍略偏 Demo 风格。
2. 页面层级关系偶尔会显得复杂。
3. 流转图、岗位分析卡片、时间轴还可以继续统一视觉语言。

### 8.5 一些能力是“第一版成立”，不是“最终版打磨”

比如：

1. 折叠准确度
2. 关联深度
3. 数据源兼容性
4. 大文档稳定性
5. AI 输出细腻度

这些都已经有了方向，但没有做到最强态。

## 9. 当前版本最值得讲的完成点

如果要面向评审或录视频，当前最值得强调的不是“功能堆了多少”，而是以下几件事：

1. **原文不变，角色视图变化**：这是区别于普通 AI 改写文档的关键点。
2. **版本确认后才触发 AI 结论**：这让 AI 参与的是“决策溯源”，不是随手生成。
3. **AI 能力按模块拆分**：分析、审阅、折叠、对话、版本、可视化、关联都是独立服务。
4. **最终编辑页承载了文档、角色、版本、AI 对话、代码、数据、关联文档**：已经具备“无界复合文档”的雏形。

这 4 条是当前 Demo 的核心价值，不建议在表达时被一些边角 UI 问题冲淡。

## 10. 后续还需要多少时间

这里分成两档判断。

### 10.1 如果目标是“比赛提交前收口”

预计还需要：

1. UI 细节统一与视觉收口：`0.5 - 1 天`
2. AI prompt 再收口，提升分析具体度：`0.5 天`
3. 典型演示文档与预置数据再打磨：`0.5 天`
4. 全链路回归验收与录屏准备：`0.5 天`

合计：

`2 - 2.5 天`

### 10.2 如果目标是“进一步逼近 PRD 理想态”

预计还需要：

1. 深化关联文档与知识网络：`1 - 2 天`
2. 提升角色折叠准确率与解释性：`1 - 2 天`
3. 加强时间轴版本对比与历史感：`1 天`
4. 提升数据接入与图表真实度：`1 - 2 天`
5. 增强多人协作或审批语义表达：`2 - 4 天`

合计：

`6 - 11 天`

## 11. 当前完成度结论

如果按“PRD 主线是否成立”来看，当前项目已经完成了大约 **75% - 80%**。

如果按“比赛 Demo 是否够用”来看，当前项目已经达到 **可以展示、可以讲完整故事、可以支撑评分亮点** 的状态。

如果按“是否完全达到 PRD 理想产品态”来看，还没有完成，主要差在：

1. AI 输出质量稳定性
2. 无界能力深度
3. 协作真实性
4. UI 最终成品质感

因此，更准确的结论应该是：

> 当前版本已经完成了 PRD 主线闭环和核心亮点验证，属于“可交付 Demo + 明显可见的下一步空间”，而不是“全部功能完全产品化”。

建议外部表达时，不要说“我们已经做完了一个完整的 AI 协作文档平台”，而是说：

> 我们已经把 FlowDoc 的核心主线做成了可运行 Demo：从工作流分析、角色视图、审阅折叠、AI 对话、版本时间轴到无界能力扩展，都已经形成闭环。当前版本重点验证的是“文档如何从静态记录工具升级成 AI 原生决策空间”。部分能力已经可跑通，部分能力以 Demo 级形态落地，后续会继续增强深度与稳定性。

这个说法会更真实，也更稳。

## 13. 最终结论

当前这版 FlowDoc：

1. 已经不是单纯的 Markdown 编辑器改造。
2. 已经形成了“工作流分析 + 角色审阅 + 决策溯源 + 无界扩展”的核心产品骨架。
3. 已经具备比赛级 Demo 的完整演示主线。
4. 但距离 PRD 中更强的产品理想态，仍有一段明确且可量化的提升空间。

因此，这个项目现在最合适的定位是：

> **一个已经完成核心闭环验证、具备较强展示力、但仍需进一步收口和深化的 AI 原生无界文档 Demo。**
