# 智能周报助手技术方案说明

## 文档信息

- 文档版本：v1.1
- 更新时间：2026-05-17
- 负责人：后端负责人 / 前端负责人
- 对应产品需求：智能周报助手产品需求说明 v0.9
- 当前目标：支撑 Demo Phase 1 验收与后续接入

## 一、方案目标

本方案用于支撑“智能周报助手”在网页端完成以下能力：

- 原始 Markdown 文档编辑
- 工作流选择与岗位流转分析
- AI 生成岗位摘要、关注重点与任务安排
- 保留完整原文，不将 AI 内容直接写回正文

技术方案的核心原则如下：

- 原文是唯一事实源，AI 结果只做旁路分析
- 工作流配置、首次分析、编辑体验三者分离
- AI 输出必须结构化、可校验、可回退
- 现阶段优先保证演示闭环，而不是过早引入数据库和协同能力

## 二、整体架构

### 2.1 页面层

- `/`：首页，展示 FDoc 品牌与演示入口
- `/setup`：配置与分析页，负责工作流、文档来源、首次 AI 分析
- `/editor`：编辑页，负责完整原文编辑与 AI 辅助信息展示

### 2.2 服务层

- FastAPI 负责路由、接口和模板渲染
- Jinja 负责页面骨架输出
- 静态资源目录负责样式、交互和 OverType 初始化
- Qwen 分析服务负责提示词组装、请求模型、清洗 JSON、结构校验

### 2.3 资产复用策略

- 复用 `overtype.min.js` 作为 Markdown 编辑器内核
- 复用 `diagram-embed.html` 作为配置页顶部的静态展示区
- 不修改 OverType 核心源码，业务逻辑全部放在 FDoc 自己的模板和脚本中

## 三、关键设计原则

### 3.1 原文与 AI 结果彻底分离

正文编辑区只存储和展示用户原始 Markdown。AI 返回的内容包括：

- 文档摘要
- 岗位流转图
- 各岗位任务与重点关注项
- 工作任务安排

这些内容只允许渲染在编辑器外部区域，不能覆盖原文、不能裁剪原文，也不能把 AI 内容写回正文。

### 3.2 工作流先于编辑区

用户必须先完成：

1. 选择工作流
2. 选择文档来源
3. 触发首次分析

分析成功后才允许进入编辑页。这样做是为了保证编辑页里展示的 AI 结果有完整上下文，也能避免用户在编辑时被再次配置打断。

### 3.3 AI 输出必须结构化且顺序稳定

岗位顺序必须完全遵守用户选择，模型不能自行重排。否则：

- 前端无法稳定渲染岗位流转图
- 岗位卡片无法与任务安排一一对应
- 后续“按岗位折叠正文”会失去语义基础

## 四、前端设计说明

### 4.1 首页

- 页面视觉尽量延续现有 OverType 首页的黑色极简风格
- 顶部品牌改为纯文字 `FDoc`
- 底部固定保留 `View Demo` 按钮，作为进入 `/setup` 的主入口

### 4.2 配置与分析页

页面由三块组成：

- 顶部静态流程图示意区
- 中部工作流配置与文档导入区
- 底部 AI 分析结果区

交互要求：

- 工作流区使用抽屉展开，自定义岗位支持排序
- 预置文档要有明显的信息量展示，而不是只有标题
- 点击“开始分析”前必须完成工作流和文档选择
- 分析成功后才出现“进入编辑区”

### 4.3 编辑页

- 页面顶部展示文档名、当前工作流与返回配置页入口
- 中部放 OverType 编辑器壳层，保留较大的左右留白和少量上下留白
- 下方展示文档摘要与岗位卡片
- 右侧使用覆盖式任务安排抽屉，不直接挤压正文可用空间

## 五、状态与数据流

### 5.1 浏览器会话状态

配置页和编辑页之间通过 `sessionStorage` 共享以下数据：

- `documentName`
- `documentContent`
- `workflow`
- `sourceType`
- `analysis`

这样做的原因：

- 实现简单，适合当前演示场景
- 刷新页面后仍能恢复最近一次状态
- 不需要数据库即可闭环验证

### 5.2 文档来源策略

当前只支持两种文档来源：

- 本地上传：`.md`、`.txt`
- 仓库内预置文档：产品、技术、数据复盘三类

预置文档除了正文外，还需要返回：

- 预览摘要
- 推荐工作流
- 内容标签
- 文档复杂度
- 章节数量

这样用户在 `/setup` 就能感知这不是“示例占位文档”，而是真实业务材料。

## 六、后端接口说明

### 6.1 健康检查

- `GET /api/health`
- 返回服务状态、千问是否已配置、当前模型名

### 6.2 预置文档列表

- `GET /api/presets`
- 返回标题、说明、预览、标签、推荐工作流、字符数和章节数

### 6.3 预置文档详情

- `GET /api/presets/{preset_id}`
- 返回完整正文和同一份元信息，供配置页载入

### 6.4 AI 分析接口

- `POST /api/analyze`
- 输入：文档名称、正文、工作流、来源类型
- 输出：摘要、岗位流转图、岗位卡片、任务安排

## 七、AI 调用与校验策略

### 7.1 提示词约束

模型提示词必须明确要求：

- 只返回纯 JSON
- 不允许返回 Markdown 代码块
- `role_flow.stages` 与 `roles` 必须和工作流一一对应
- 任务安排必须覆盖整个工作流
- `view_hints` 仅作为未来折叠能力保留字段

### 7.2 输出校验

后端执行以下校验步骤：

1. 检查返回文本是否为空
2. 尝试直接解析 JSON
3. 若失败，清理代码块标记和前后多余文本
4. 再次解析并交由 Pydantic 模型校验
5. 若仍失败，触发一次修复性重试
6. 最终仍失败则返回结构化错误对象

### 7.3 顺序一致性检查

除了 Pydantic 校验外，还需要对以下内容做二次比对：

- `role_flow.stages[*].role`
- `roles[*].role`
- `task_schedule[*].owner`

如果岗位名称集合或顺序与输入工作流不一致，接口应拒绝该次分析结果。

## 八、接口协议示例

### 8.1 请求体

```json
{
  "document_name": "需求说明.md",
  "document_content": "# 标题\n\n正文",
  "workflow": ["产品经理", "工程师", "数据分析员", "CEO"],
  "source_type": "upload"
}
```

### 8.2 返回体核心字段

```json
{
  "document_summary": "一句到两句的整体摘要",
  "role_flow": {
    "title": "岗位流转图",
    "stages": [
      {
        "role": "产品经理",
        "stage_goal": "明确需求范围与验收标准",
        "handoff_to_next": "向工程师移交需求边界、优先级和验收口径"
      }
    ]
  },
  "roles": [],
  "task_schedule": []
}
```

## 九、测试与上线策略

### 9.1 本地验证

- 使用 `py_compile` 检查 Python 语法
- 使用 `TestClient` 验证关键路由和接口可访问
- 在浏览器中完成首页、配置页、编辑页的主流程验收

### 9.2 重点风险回归

- 预置文档选择后能否正确写入会话状态
- 直接访问 `/editor` 是否会被正确重定向
- 分析失败时是否意外保留旧的岗位流转图
- 任务抽屉打开时是否影响编辑器可用宽度

### 9.3 上线建议

- 首先保证演示环境 API Key 已配置
- 预置文档优先使用结构完整、岗位感强的真实材料
- 每次前端结构修改后都重新验证 `/setup` 与 `/editor` 的 session 恢复逻辑

## 十、后续扩展方向

### 10.1 内容层

- 增加更多岗位组合模板
- 增加更多真实业务预置文档
- 支持项目复盘、事故复盘、方案评审等不同文档类型

### 10.2 能力层

- 接入数据库保存历史分析快照
- 支持在编辑页中重新分析
- 支持按岗位折叠正文，但必须保证原文不可被改写

### 10.3 协作层

- 对接企业消息通知
- 生成岗位待办摘要导出
- 将任务安排同步到项目管理工具
