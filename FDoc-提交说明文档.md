## 模块一 用户洞察与问题定义

### 目标用户

FDoc 面向的是围绕同一份业务文档协作的多角色团队，而不是单一写作者。典型用户包括：

1. 产品经理：需要用一份文档推动需求评审、范围确认、协作分工和里程碑对齐。
2. 工程师：需要快速定位与实现相关的接口、边界、风险和交付信息。
3. 数据分析员：需要围绕指标、波动原因、ROI、复盘结论进行审阅与补充。
4. CEO / 负责人：需要在有限时间内快速理解文档的关键结论、决策风险和下一步动作。

这些用户共同面对同一份原文，但每个角色真正关心的信息不同。FDoc 的核心目标，是让同一份文档在不被改写的前提下，按角色呈现不同的审阅视角。

### 用户痛点

当前企业文档协作中常见的问题包括：

1. 文档内容面向所有人平铺展示，信息密度高，但角色相关性低。
2. 产品、工程、数据、管理者阅读同一份文档时，往往要各自手动筛选重点，重复成本高。
3. 文档修改后只留下内容变化，缺少“为什么改、谁受影响、最后为什么这么定”的决策解释。
4. AI 在许多文档工具里主要承担润色或改写，容易改变原文事实源，反而增加协作不确定性。
5. 代码、数据、关联资料、审阅意见分散在不同工具中，文档无法承接完整工作上下文。

### 使用场景

FDoc 聚焦一个高频场景：团队围绕一份项目文档进行跨岗位流转、审阅、修改和决策沉淀。

典型使用流程如下：

1. 用户在配置页选择工作流角色，并上传 `.md/.txt` 文档或选择预置文档。
2. 系统基于原文与工作流，生成文档摘要、岗位任务、角色关注点和流转信息。
3. 进入编辑页后，默认展示左侧对话栏与中间完整原文，便于边读边问。
4. 用户切换角色后，可在普通编辑区生成该角色的“折叠审阅”视图：重要内容展开，弱相关内容折叠，且所有内容仍来自同一份原文。
5. 用户点击 `AI 副驾` 后进入三栏工作台：左侧副驾问答，中间原文，右侧可编辑 AI 结果稿。
6. 用户确认修改后，系统生成版本摘要、阶段结论和决策溯源记录，形成可回看的版本时间轴。

## 模块二 产品方案设计

### 产品概述

FDoc 是一个 AI 原生的无界文档协作工作台。它不是把聊天机器人简单放到 Markdown 编辑器旁边，也不是让 AI 直接替用户改写正文，而是把文档升级为一个可以被多角色理解、审阅、追问、补充和溯源的业务空间。

FDoc 的核心设计原则是：

1. 原文是唯一事实源，AI 不自动覆盖原文。
2. 角色视角只改变阅读与审阅方式，不制造多份互相冲突的正文。
3. AI 参与工作流分析、折叠审阅、计划生成、审阅稿生成和版本总结，而不只是润色文本。
4. 文档工作台同时承载对话、代码实验、图表试验、文档关联和版本时间轴。

### 核心功能

#### 1. 工作流配置与文档分析

FDoc 支持用户在进入编辑前完成工作流配置：

1. 支持默认工作流和自定义角色工作流。
2. 支持上传 `.md/.txt` 文档或选择系统预置文档。
3. AI 根据原文内容与角色顺序生成文档摘要、角色任务、关注点和工作流分析结果。
4. 分析结果写入本地会话状态，用于后续编辑页的角色切换、对话和折叠审阅。

#### 2. 普通编辑区：左侧对话 + 完整原文

编辑页默认是普通编辑态：

1. 左侧常驻 AI 对话栏，可持续围绕当前文档提问。
2. 中间保留完整 Markdown 原文编辑器，用户可以直接修改原文。
3. 用户选中原文片段后，可把片段加入左侧对话上下文。
4. 页面不默认进入三栏副驾模式，避免干扰普通文档阅读和编辑。

#### 3. 角色折叠审阅

普通编辑区中提供“角色审阅”能力：

1. 用户切换到不同岗位角色后，可点击生成该角色的折叠审阅视图。
2. AI 根据当前角色任务、关注点、关键词和原文章节，判断每段内容的相关性。
3. 与当前角色强相关的段落默认展开并高亮，弱相关段落默认折叠。
4. 折叠只影响视图层，不删除、不改写、不覆盖原文。
5. 用户可以手动展开或收起每一个折叠段落，保证演示和实际使用都可控。

#### 4. AI 副驾三栏工作台

点击左侧 `AI 副驾` 后，页面进入三栏工作台：

1. 左栏：副驾面板，包含 `Plan` 与 `Critique`。
2. 中栏：完整原文编辑区，仍保持可读可改。
3. 右栏：可编辑 AI 结果稿编辑器，用户可以继续修改 AI 生成内容。

当前副驾包含两类能力：

1. `Plan`：支持逐题问答生成计划草稿，也支持直接点击生成完整计划。
2. `Critique`：输入审阅目标后，左侧生成问题列表，右侧生成可编辑审阅稿。

右侧结果稿与中间原文默认独立，只有用户主动选择文本并确认写回时，才会追加到原文目标章节。

#### 5. 版本时间轴与决策溯源

FDoc 不把每次输入都自动视为正式版本，而是采用“确认更改”机制：

1. 用户修改原文后，页面标记为待确认状态。
2. 点击 `确认更改` 后，系统生成新版本。
3. AI 自动生成变更摘要、自写结论、关键变化和受影响角色。
4. 左侧抽屉展示版本时间轴，支持回看和恢复历史版本。

#### 6. 无界能力扩展

普通编辑区保留多类文档外能力：

1. 代码实验区：支持 HTML / JavaScript 浏览器预览，并支持 Python / C / Java 的 AI 解释与伪运行结果。
2. 图表试验区：支持粘贴轻量数据，生成表格、图表建议和字段说明。
3. 文档关联：基于预置文档推荐关联资料，并支持关系说明、关系图和只读预览。

这些能力只出现在普通编辑区，不会混入 AI 副驾三栏工作台。

### 产品架构 / 功能图

FDoc 当前 Demo 可概括为五层架构：

1. 文档输入层：上传文档、预置文档、工作流配置。
2. AI 分析层：文档分析、角色审阅增强、折叠规划、计划生成、审阅稿生成、版本总结。
3. 文档编辑层：左侧对话、完整原文编辑、角色切换、折叠审阅。
4. 副驾工作台层：Plan 问答与完整计划、Critique 审阅稿、右侧可编辑 AI 结果稿。
5. 无界扩展层：代码实验、图表试验、文档关联、版本时间轴。

可以用下面的流程理解产品结构：

```text
配置工作流 / 导入文档
        ↓
AI 文档分析与角色理解
        ↓
普通编辑态：对话栏 + 完整原文 + 角色折叠审阅
        ↓
AI 副驾态：Plan / Critique + 右侧可编辑结果稿
        ↓
用户确认写回或确认更改
        ↓
版本时间轴与决策溯源
```

### 交互流程

1. 用户进入配置页，选择工作流角色并导入文档。
2. 点击开始分析，系统生成文档摘要、角色任务和工作流信息。
3. 用户进入编辑页，默认看到左侧对话栏和中间完整原文。
4. 用户可以切换角色，生成该角色的折叠审阅视图。
5. 用户需要更强 AI 协助时，点击 `AI 副驾` 进入三栏工作台。
6. 在 `Plan` 中，用户可以逐题回答并生成计划草稿，也可以直接生成完整计划。
7. 在 `Critique` 中，用户输入审阅目标，得到问题列表和可编辑审阅稿。
8. 用户需要把 AI 结果写回原文时，必须在右侧选中文本并确认目标章节。
9. 用户修改原文后，点击 `确认更改`，系统生成版本记录和 AI 总结。

### 创新与差异化

FDoc 的差异化主要体现在四点：

1. 原文不变，视角变化：不同角色看到的是同一份原文的不同折叠审阅视图，而不是 AI 改写出的多个版本。
2. AI 参与协作流程，而不是只做润色：AI 覆盖分析、折叠、计划、审阅、版本总结和关联推荐。
3. 副驾结果稿独立可编辑：AI 输出先进入右侧结果稿，用户确认后才写回原文。
4. 文档成为复合工作台：同一页面内聚合对话、代码、数据、关联资料、版本溯源和副驾工作流。

## 模块三 AI 原生能力说明

### AI 核心能力

FDoc 当前接入的 AI 能力包括：

1. 文档与工作流联合分析：根据文档内容和角色顺序生成岗位任务、摘要和关注点。
2. 角色审阅增强：为每个角色生成审阅摘要、审阅清单和视图提示。
3. 角色折叠规划：判断原文章节对当前角色的相关性，给出高亮、保留或折叠建议。
4. AI 对话：结合当前文档、角色和用户选中的上下文片段回答问题。
5. Plan 副驾：根据原文、角色和问答历史生成计划草稿或完整计划。
6. Critique 副驾：根据审阅目标生成结构化问题列表和可编辑审阅稿。
7. 版本总结：在用户确认更改后生成变更摘要、阶段结论和受影响角色。
8. 无界辅助：为代码实验、图表试验和文档关联提供解释、推荐和补全。

### AI 如何解决痛点

FDoc 用 AI 解决的不是“如何把文字写得更漂亮”，而是“如何让多人围绕同一份文档更高效地理解、审阅和决策”：

1. 对信息过载问题，AI 按角色生成折叠审阅视图，让每个岗位先看重要内容。
2. 对协作分工模糊问题，AI 根据工作流提取角色任务、关注点和审阅清单。
3. 对修改过程难以追溯问题，AI 在确认更改后生成版本摘要和阶段结论。
4. 对 AI 自动改写带来的不确定性，FDoc 让 AI 输出进入独立结果稿，原文写回必须由用户确认。
5. 对文档与代码、数据、资料割裂的问题，AI 把代码解释、图表建议和关联文档推荐纳入同一工作台。

### AI 技术方案

当前 Demo 采用轻量但可运行的技术方案：

1. 前端：Jinja 模板 + 原生 JavaScript + OverType Markdown 编辑器。
2. 后端：FastAPI 提供页面路由、预置文档接口和 AI 能力接口。
3. 大模型：通过千问 / DashScope API 调用，使用 `DASHSCOPE_API_KEY` 配置。
4. 状态管理：编辑页状态主要保存在 `sessionStorage`，副驾内容按角色分桶保存。
5. 双编辑器：中间原文编辑器与右侧副驾结果稿编辑器独立运行。
6. 结构化输出：AI 返回统一要求 JSON，再通过 Pydantic 校验，必要时进行 JSON 修复或兜底处理。
7. 能力拆分：分析、审阅增强、折叠规划、对话、副驾、代码、图表、文档关联、版本总结分别拆成独立服务，降低单次调用复杂度。

### 当前 Demo 的稳定性处理

为了保证比赛演示可运行，FDoc 在关键 AI 能力中加入了兜底策略：

1. AI 返回 JSON 不规范时，后端会尝试修复结构。
2. 代码实验、图表试验、文档关联和折叠规划均保留 fallback 结果。
3. 右侧副驾结果稿被用户手动修改后，再次由 AI 覆盖前会提示确认。
4. 版本时间轴只在用户点击 `确认更改` 后生成，避免演示中产生混乱记录。

## 模块四 加分项（可选）

### 落地可行性

FDoc 适合从企业内部高频文档协作场景切入，例如：

1. 产品需求评审文档
2. 技术方案评审文档
3. 数据复盘与经营周报
4. 多角色联合决策备忘录
5. 项目阶段总结与版本决策记录

这些场景共同特点是：文档信息密度高，参与角色多，阅读目标不同，并且需要留下决策依据。FDoc 的“原文保留 + 角色折叠 + AI 副驾 + 版本溯源”正好对应这类需求。

从工程落地角度看，当前 Demo 已经具备清晰的接口拆分和可扩展结构。后续可以逐步接入企业知识库、权限系统、多人协作、真实数据源和文档审批流。

### 商业化思考

FDoc 可以沿两条路径发展：

1. 企业内部部署：面向产品、研发、数据和管理团队，作为 AI 原生协作文档工作台。
2. SaaS 服务：按团队席位、AI 调用次数、文档空间容量或高级协作能力收费。

潜在商业价值包括：

1. 降低跨角色文档阅读成本。
2. 提升需求评审、方案评审和经营复盘效率。
3. 沉淀可追溯的版本结论和决策依据。
4. 将文档从“记录工具”升级为“AI 协作入口”。
